# Rare Disease Gene Hackathon 项目管理与执行工具包

这是一个用于筹备、执行、复盘和交接「罕见病基因黑客松」的 SOP 与协作工具包。

它把一次真实活动中的组织经验沉淀为可复用的项目手册、分组执行指南、风险预案、工具表和授权文件，帮助后续团队更安全、更清楚地组织未确诊罕见病相关的跨学科协作。

## 30 秒了解

**这个项目是什么？**

一个面向罕见病基因黑客松筹备团队的开源文档工具包，覆盖项目统筹、数据与技术、赛手、嘉宾评审、会务、传播、设计、财务和风险管理。

**为什么存在？**

罕见病基因黑客松涉及真实病例、基因数据、医学专家、赛手团队、现场执行和公众传播。如果缺少统一 SOP，项目容易在数据安全、医学边界、跨组协作和赛后沉淀上失控。本工具包用于降低重复组织成本，并让经验可以被下一期复用。

**适合谁使用？**

- 正在筹备罕见病基因黑客松或类似公益科技活动的项目统筹团队
- 负责病例、基因数据、赛手、专家、会务、传播、设计或财务的工作组负责人
- 希望理解这类活动如何组织、如何控风险、如何沉淀成果的协作者
- 需要快速理解项目边界和文档结构的 AI Agent

## 重要边界

本工具包用于项目管理、协作执行和活动复盘，不用于提供临床诊断、医疗替代方案或个案治疗建议。

涉及患者、病例、基因数据和医学结论的内容，必须遵守授权、脱敏、医学审核和数据安全要求。任何公开传播都不能暴露患者隐私，不能把分析线索包装成诊断结论。

## 推荐阅读路径

### 如果你是第一次打开这个仓库

1. 先读 `01_README.md`，确认项目用途、边界和目录结构。
2. 再读 `02_筹备手册/01_项目总统筹.md`，理解项目目标、成功标准、分工和决策机制。
3. 接着读 `02_筹备手册/03_风险管理与应急预案.md`，确认哪些事情必须升级处理。
4. 最后根据自己的角色进入对应小组手册和工具表。

### 如果你是项目统筹或发起人

1. `02_筹备手册/01_项目总统筹.md`
2. `02_筹备手册/03_风险管理与应急预案.md`
3. `02_筹备手册/02_各小组工作手册/01_发起人与总负责人工作手册.md`
4. 所有小组工作手册
5. `03_系列工具表/01_发起人与总负责人_工具表.xlsx`

统筹的重点是维护项目边界、关键节点、跨组依赖、决策记录和风险升级机制。

### 如果你是工作组负责人

1. `02_筹备手册/01_项目总统筹.md`
2. `02_筹备手册/03_风险管理与应急预案.md`
3. 本组工作手册
4. 本组 Excel 工具表
5. 与本组有强依赖的小组手册

例如，数据与技术组应重点阅读 `02_数据与技术组工作手册.md`，同时了解嘉宾与评审组如何确认病例、样本关系、参考基因组版本和医学字段。

### 如果你是志愿者或临时加入成员

不需要阅读全部文档。请先确认自己属于哪个小组，再阅读：

1. 所属小组工作手册
2. `02_筹备手册/03_风险管理与应急预案.md` 中与现场、数据、传播相关的部分
3. 需要签署或执行的协议、授权文件

你必须清楚三件事：自己负责什么、遇到异常找谁、哪些信息不能传播。

### 如果你是 AI Agent

建议按以下顺序建立上下文：

1. 读取 `01_README.md` 获取项目边界、目录结构和推荐路径。
2. 读取 `02_筹备手册/01_项目总统筹.md` 获取项目定义、原则、角色边界和决策机制。
3. 读取 `02_筹备手册/03_风险管理与应急预案.md` 获取风险等级、升级规则和应急处理逻辑。
4. 仅在任务涉及具体小组时读取对应小组手册和工具表。
5. 不要把 `04_初代团队手记/` 当作执行规则；它主要用于理解背景和叙事。

处理本仓库时，应优先保持文档间的一致性：编号、目录名、角色名称、风险等级、工具表名称和医学/数据边界不要彼此冲突。

## 如何开始使用

1. 复制本工具包作为本期活动工作目录。
2. 先更新 `02_筹备手册/01_项目总统筹.md` 中的项目目标、时间地点、主办方、成功标准和关键负责人。
3. 为每个工作组指定负责人，并让负责人阅读对应手册。
4. 从 `03_系列工具表/` 复制需要使用的 Excel 和 Word 文件，作为本期活动工作底表。
5. 在活动前确认数据、医学、传播和现场风险是否已有负责人和升级路径。
6. 活动结束后，把有效流程、踩坑记录和复盘结论更新回手册或工具表。

不建议直接覆盖模板原件。每一期活动应保留独立副本，便于复盘和版本追踪。

## 文档维护原则

- README 只承担入口和导航功能，不承载全部执行细节。
- 稳定规则写入手册，临时事项写入本期活动记录。
- 工具表字段变化后，应同步检查对应小组手册是否需要更新。
- 涉及数据安全、医学边界、授权条款、评审规则和公开口径的修改，必须经过项目统筹确认。
- 新增文件前先确认：谁使用、何时使用、输入是什么、输出是什么、由谁维护。

## 项目原则

- 患者优先：组织、传播、评审和成果沉淀都不得让患者承担额外风险。
- 数据安全：病例资料、基因数据、患者信息和内部分析结果必须按最小必要原则访问、存储和分发。
- 科学严谨：分析结果应保留证据链、方法说明、不确定性和限制条件。
- 开源协作：可复用流程、工具、字段结构和项目经验应尽可能开放；原始数据和可识别个人的信息不得开放。
- 长期复用：每期活动都应为下一期降低组织成本，形成可交接、可复盘、可迭代的资产。
